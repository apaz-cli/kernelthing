## What a problem is
kernelthing runs a loop that repeatedly edits a kernel and scores it with
pygpubench: the submission is imported **by qualified name** in an isolated,
sandboxed subprocess and timed against a reference test case. To make that
possible you must author:

1. **`problem.json`** — the manifest. Follow this schema (a real example):

   The `//` notes are explanatory — emit **valid JSON** (no comments):

   ```jsonc
   {
     "name": "<short-slug>",
     "description": "<one sentence: kernel, shapes, hardware>",
     "plan": "plan.md",                            // path to the plan.md you write
     "edit_files": ["kernel.py"],                  // the kernel source — NEVER submission.py
     "bench": {
       "submission_qualname": "submission.kernel"  // adapter module.func that calls the kernel
     },
     "metric": {                                   // how the median time becomes the score
       "kind": "pct_baseline",                     // pct_baseline | speedup | tflops | latency_us
       "baseline_qualname": "baseline.kernel"      // denominator (pct_baseline/speedup); tflops needs "flops"
     },
     "metric_name": "pct_baseline",                // short label for logs/UI
     "unit": "%cuBLAS",                            // display unit
     "direction": "maximize"                       // pct_baseline/speedup/tflops: maximize; latency_us: minimize
   }
   ```

2. **`plan.md`** — a short plan describing the goal, the target hardware,
   the problem shape/dtype, the correctness reference, and the optimization
   objective. The loop reads this every round. Keep it small, concrete,
   and concise, but retain the relevant considerations to the problem. When
   interactive, confirm the plan and every input/output shape and dtype with the
   operator before proceeding (see *Operating mode* above).

   It **must** include a `## Build & test` section telling the optimizer how to
   build and score the kernel — the single authoritative command, with the actual
   problem dir filled in:

   ```
   ## Build & test
   Build, run, and score the kernel with the authoritative benchmark:

       kernelthing score {{TARGET_DIR}}

   It prints `{"correct": ..., "metric": ..., "unit": ...}` — the same scoring
   the loop uses every round. `"correct": true` with a sane metric means the
   edit builds and passes correctness; read the `error` field on failure. (From
   inside the problem dir, `kernelthing score .` is equivalent.)
   ```

   There is no separate build step or test harness to document — `kernelthing
   score` compiles/loads the kernel and checks correctness itself. Do not invent
   a different build/test command.

3. **`task.py`** — define `generate_test_case(*, seed)` returning a tuple
    `((inputs...), (expected, atol, rtol))`. Bake the problem's shapes/dtypes
    directly into this function (no external args):
    - The first element is the tuple of arguments the kernel is called with.
    - `expected` is the correct output (compute it with a trustworthy reference,
      e.g. a torch op);
    - `atol`/`rtol` are the absolute/relative correctness tolerances.
    - Be sure to move input tensors onto the device before returning.

    **Contiguity.**  Every tensor in the inputs tuple should probably be
    contiguous (`.is_contiguous() == True`).  pygpubench's tensor conversion
    faults on non-contiguous bf16 tensors, and the kernel receives raw
    `.data_ptr()` pointers whose physical memory layout must match what
    the kernel logic expects. Prefer row-major tensors (the natural
    contiguous layout in PyTorch) and write your kernel to match unless
    otherwise specified.

    **Natural signatures.**  The kernel and `generate_test_case()` should
    use natural Python signatures. The kernel *returns* results, and the
    generator returns only the logical inputs.

4. **`submission.py`** is the boundary between pytorch and the kernel. A small shim.
   It should import the kernel (via python import or pytorch's `load_inline()`
   or similar), and call it on the inputs in a public function named by
   `submission_qualname`. This function accepts exactly the inputs generated by 
   `generate_test_case()`, calls the kernel, and returns the outputs. Format conversion
   should not be performed here. Note that `submission.py` is NEVER an `edit_file`.
   Edits happen only to the kernel source, this is only a small shim on top to make
   it callable from python.

5. **the kernel source** (the single `edit_file`, e.g. `kernel.py`, `kernel.cu`, etc).
   A correct and simple starting implementation of the kernel, which `submission.py` calls.
   This is the *only* file the optimizer edits.

6. **`baseline.py`** (only if `metric.kind` is `pct_baseline`) — a copy of
   `submission.py`, but importing and calling the reference kernel (item 7)
   instead of the editable one, in a public function named by `baseline_qualname`.
   Like `submission.py` it is NEVER an `edit_file`.

7. **the reference kernel source** A reference kernel of the same signature, which
   `baseline.py` calls. Used for numerical correctness, and to measure speedups.
   Usually this means a copy of the kernel source, but it could also be something like
   a torch or cuBLAS op, for example. It is best that this be a straightforward and
   trustworthy implementation. Do not tune this, pick launch parameters, or
   anything of that sort. It is for comparison, and it is NEVER an `edit_file`.

## Kernel language
The submission is always reached through the fixed `submission.py` adapter, but the
kernel source it calls — the one `edit_file` — can be in any language:

- **PyTorch / Triton / CuteDSL / numba** — the kernel is Python (e.g. `kernel.py`,
  holding any `@triton.jit` / `@cute.kernel`); `submission.py` imports and calls it.
- **Raw CUDA `.cu` / C++** — the kernel is the `.cu`; `submission.py` compiles +
  loads it (e.g. `torch.utils.cpp_extension.load`, or `nvcc` + `ctypes`/`cffi`) and
  calls it.

When using `load_inline` with a `.cu` kernel: put only a forward declaration of your
entry function in `cpp_sources` (no `#include <torch/extension.h>`, no `PYBIND11_MODULE`),
put the full definition plus the kernel in `cuda_sources`, and list the function name in
`functions`. Anything beyond that collides with what `load_inline` auto-generates.

Build artifacts (nvcc temp files, load_inline output) must land inside the problem
directory. The benchmark sandbox blocks writes everywhere else. Set build_directory
to a path under the problem dir and point TMPDIR/TMP/TEMP there before calling load_inline.

Keep `edit_files` **as minimal as possible** — ideally just that single kernel
source. Every extra editable file widens the cheat surface and dilutes the
optimizer's focus. `submission.py`, `task.py`, `baseline.py`, and the reference
kernel must never be in `edit_files`.

Derive shapes/dtypes/tolerances from the objective. Do not guess silently — if
the objective is ambiguous, choose the most defensible interpretation and state
it in your summary.

**Getting the original and reference kernels right are the single most important part of this task.**
Both the `expected` output in `task.py` and the reference kernel (item 7) are
*references*: they define what "correct" means for every later round, and a wrong
reference silently corrupts the entire run while everything still looks green. So
when you produce a reference, do everything you can to be very sure it is
numerically correct. It's best to cross-check it against another implementation if
one is available. If you cannot establish a reference you are confident in, say
so rather than shipping one you are unsure about.

## Self-test
Before finishing, confirm the objective actually runs and the shipped submission
is correct: from inside `{{TARGET_DIR}}` run

    kernelthing score .

and confirm it prints `"correct": true` with a sane metric. Also **verify
`direction`**: a faster kernel must move the metric the way `direction` rewards; if
not, fix `direction` before finishing. This is the exact
`bench.score` call that gates this bootstrap turn and scores every loop round —
it benchmarks the submission through pygpubench, checks it against `expected`
within `atol`/`rtol`, and (for `pct_baseline`/`speedup`) does the same for
the reference kernel. So a green here is the real verdict, with no separate
self-test harness to drift out of sync. You do not need to author a test file. If
you need to debug a failure, run `kernelthing score .` and read its `error` field.

## Finishing
- Do not finish unless `kernelthing score .` printed `"correct": true`. This is
  very important.
- If you fully authored a valid, self-tested problem, end your message with a
  short "where things are" summary (the problem directory path, the manifest's
  name, the input shapes, the tolerances, the submission binding, the metric)
  and then the single word `{{COMPLETE}}` on its own line.
- Only when an objective **was** provided but is genuinely impossible to realize
  (truly unknowable reference/shapes that you cannot defensibly resolve), explain
  precisely what is missing and end with the single word `{{SETUP_BLOCKED}}` on its
  own line. Do not fabricate a plausible-but-wrong objective — a wrong objective
  silently corrupts every later round.
